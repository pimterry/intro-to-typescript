TypeScript

JavaScript + types

  • Anyway, what is TypeScript
  • Compile to JS language
  • JS superset
  • Type checked
  • Optional typing, unlike Java etc
  • Inferred types: automatically detected wherever possible, just plain JS
  • There's a reason why this is useful

"Dynamic typing": The belief that you can't explain to a computer why your code works, but you can keep track of it all in your head.

— Chris Martin (@chris__martin) August 10, 2015
  • There's a reason this is useful: dynamic typing doesn't scale well
  • Your code still has types: this function takes an object with a 'name' property
  • But they're not written down. You have to know them, or go look them up
  • You're not that good at that. The computer is.
  • Static types let you leave the computer to that, and use its knowledge, rather than having to know everything yourself
  • Computer understanding this is like having guaranteed up to date documentation at your fingertips
  • Sort of like unit testing, but from the other direction: guaranteed you can't make whole classes of errors (rather than guaranteed that specific cases are ok)
  • Example, to try and convince you
navigator.geolocation.getCurrentPosition(function onSuccess(position) {
  var lat = position.latitude;
  var long = position.longitude;
  
  var lastUpdated = Date.parse(position.timestamp);
  var now = new Date();
  var positionIsCurrent = now.getYear() === lastUpdated.getYear();
  
  if (positionIsCurrent) {
    var div = document.createElement("div");
    div.class = "message";
    div.style = "width: 100%; height: 100px; background-color: red;";		
    div.text = "Up to date position: " + lat + ", " + long;
    document.body.append(div);
  } else {
    var messageDivs = document.querySelectorAll("div.message");
    messageDivs.forEach(function (message) {
      message.style.display = false;
    });
  }
}, { enableHighAccuracy: "never" });

Type Annotation

  • var y: number;
y = 1; // compiles
y = "hello"; // won't compile: "Type 'string' is not assignable to type number"
  • function sayHelloTo(name: string): string {
  return "hello " + name;
}

var message = sayHelloTo("Tim"); // message inferred as string
var result: number = sayHelloTo("everybody"); // won't compile
message = sayHelloTo({name: "Tim"}); // won't compile
  • Need some way to specify types in the cases that doesn't cover though if you want more safety

  function Greeter() {
    this.language = "English";

    return "hi " + name;
  };
  Greeter.prototype.bye = function (name) {
    return "bye " + name;

  return Greeter;

Optional Typing

ES2015 (ES6)

  • Arrow functions (x) => x * 2
  • Destructuring var {a, b} = {a:1, b:2}
  • Spread operator var xs = [1, 2, ...iterable, 5, 6];
  • For/of for (var x of [1, 2, 3}) { ... }
  • Symbols var s = Symbol("mySymbol")
  • Let/const let x = 1; const y = "hi"
  • Template strings y = `hello ${myVar}`
  • And ES2015 modules, classes, promises, default parameters, unicode support, generators, ES7 decorators and more...
  • TypeScript also gives you pretty great ES6/2015 support
  • All big and exciting, enough for a whole separate talk
  • Three points to make around all this
  • Lots of exciting stuff here, which ties in nicely to TS (modules + classes)
  • TypeScript is ahead of the curve, can already use lots if in an environment that supports it
  • TypeScript also gives backward compatibility, so for about 30% of the spec you can use it already *right now*

TypeScript Roadmap

  • ES7 Async
  • ES7 Exponention operator (2 ** 4)
  • 'this' type
  • ES3 compat for decorators and async
  • Unreachable code detection
  • Inferring types from included .js files
  • Built-in module bundling
  • And more...
(See github.com/Microsoft/TypeScript/wiki/Roadmap for full details)

Downsides

  • Extra compile step
  • Occasional type wrangling
  • Could be more ambitious

Other options

  • Closure compiler
  • Flow
  • Babel
  • Closure compiler infers and checks types where it can
  • Limited, nowhere near as powerful or as effective
  • Can do type annotations too, but they have to be in JSDoc comments, bit painful
  • Flow is very powerful inference engine, particularly around nulls
  • Focused primarily on inference, not on adding new features
  • Also supports annotations, obviously has JSX support in too
  • Doesn't have anything like the maturity and tool support
  • No DefinitelyTyped!
  • Much weaker ES2015 support
  • Good choice if you definitely only want inference though, or need JSX *now*.
  • Babel has plugins to add type checking
  • Definitely won't have good tool support, needs more setup
  • Huge flexibility though if you're looking to glue together lots of JS extensions
